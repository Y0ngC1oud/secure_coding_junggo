# 보안 점검 및 대응 내역

개발 과정에서 고려한 보안 취약점과 각 항목에 대한 대응 방식을 정리했다.

| # | 항목 | 위험 | 대응 |
|---|------|------|------|
| 1 | 비밀번호 저장 | 평문 저장 시 DB 유출 시 계정 전체 탈취 | `werkzeug.security`의 `generate_password_hash`/`check_password_hash`(scrypt)로 해시만 저장 |
| 2 | SQL Injection | 문자열 조합 쿼리로 인한 DB 조작 | SQLAlchemy ORM만 사용, raw SQL/문자열 포매팅 쿼리 미사용 |
| 3 | XSS | 사용자 입력(채팅, 상품명/설명, 소개글 등)이 그대로 렌더링되어 스크립트 실행 | Jinja2 기본 auto-escape 유지, `|safe`/`Markup` 미사용 |
| 4 | CSRF | 로그인된 사용자가 의도치 않은 요청(송금, 삭제 등)을 실행 | Flask-WTF `CSRFProtect` 전역 적용. WTForms가 아닌 순수 HTML 폼(삭제, 관리자 조치)에도 `csrf_token()` 히든 필드 포함 |
| 5 | 브루트포스 로그인 | 무한 비밀번호 시도 | Flask-Limiter로 `/auth/login`을 분당 5회로 제한 |
| 6 | 세션 하이재킹/고정 | 세션 쿠키 탈취, JS로 쿠키 접근 | `SESSION_COOKIE_HTTPONLY=True`, `SESSION_COOKIE_SAMESITE=Lax`, 배포 환경에서는 `SESSION_COOKIE_SECURE=true`로 전환(HTTPS 전제) |
| 7 | IDOR (권한 없는 리소스 접근) | 상품 수정/삭제 URL을 직접 조작해 타인 상품 변경 | 상품 수정/삭제 시 `seller_id == current_user.id` 검증 후 아니면 403. 1:1 채팅은 자기 자신과의 대화 시도 시 404. 관리자도 본인 계정 상태는 스스로 변경 불가 |
| 8 | 파일 업로드 취약점 | 악성 파일 업로드, 경로 조작, 파일명 충돌 | 이미지 확장자 화이트리스트 검증, 원본 파일명을 신뢰하지 않고 UUID 기반 랜덤 파일명으로 저장, 전역 5MB 업로드 크기 제한(`MAX_CONTENT_LENGTH`) |
| 9 | 에러 정보 노출 | 디버그 모드로 스택트레이스/소스코드 노출 | `FLASK_DEBUG` 환경변수로 제어하며 기본값은 `false`. 운영 환경에서는 반드시 비활성 상태 유지 |
| 10 | 정지/휴면 계정의 세션 지속 | 로그인 중 관리자가 계정을 정지시켜도 기존 세션으로 계속 이용 가능 | `before_request` 훅에서 매 요청마다 계정 상태를 검사, `active`가 아니면 즉시 `logout_user()` 처리 |
| 11 | 관리자 권한 상승 | 일반 사용자가 관리자 라우트 접근 | `admin_required` 데코레이터(로그인 여부 + `role == admin` 검증), 위반 시 403 |
| 12 | 차단된 게시물 노출 | 신고로 차단된 상품이 목록/직접 URL로 계속 노출 | 목록/검색은 `status="active"`만 조회. 상세 페이지 직접 접근 시에도 판매자·관리자가 아니면 404 |
| 13 | 송금 이중 지불/잔액 무결성 | 동시 요청 시 잔액 음수화, 없는 금액 송금 | 송금 전 최신 잔액 확인 후 차감·적립·거래기록 생성을 하나의 DB 트랜잭션으로 커밋. 최소 1원 ~ 최대 1억원 검증, 본인 송금 차단 |
| 14 | 신고 기능 남용 | 동일인이 반복 신고해 인위적으로 차단 유도 | 사용자당 동일 대상 중복 신고 금지(1회 제한). 신고 수 = 서로 다른 신고자 수와 동일하게 강제되어 다수 사용자 신고가 누적됐을 때만 자동 제재(임계값 기본 3회) |
| 15 | SECRET_KEY 하드코딩 | 소스코드에 비밀키 노출 시 세션 위조 가능 | `.env`(git-ignore 처리)로 관리, 값이 없으면 실행마다 랜덤 키 생성(개발 편의용, 운영 배포 시 반드시 고정값 지정 필요) |

## 알려진 한계 (향후 개선 과제)

- 업로드 이미지의 실제 바이너리 시그니처(매직 바이트) 검증은 적용하지 않았다. 현재는 확장자 화이트리스트로만 걸러지며, 확장자를 위조한 악성 파일 업로드까지 막지는 못한다.
- Flask-Limiter가 인메모리 스토리지를 사용하므로, 다중 프로세스/서버로 수평 확장 시에는 Redis 등 외부 스토리지로 교체해야 한다.
- 운영 배포 시에는 Flask 개발 서버 대신 gunicorn 등 WSGI 서버와 HTTPS(리버스 프록시) 적용이 필요하다.
